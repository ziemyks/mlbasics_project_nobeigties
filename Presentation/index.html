The Problem

Why Fraud Detection Is Hard

Card fraud costs the industry billions of dollars every year. Detection systems are the last line of defence between criminals and customers — and a single missed transaction can wipe out a customer's savings overnight.

New credit card company, western US. Main goal:

Catch as much fraud as possible - before the customer loses money.

"When in doubt - stop the transaction.
A blocked card can be unblocked in minutes.
Money stolen is gone."

The Accuracy Trap

Only 0.52% of transactions are fraud.

99.5%
accurate - catches ZERO fraud

A model that always says "not fraud" scores 99.5% - and is completely useless. Standard accuracy cannot be trusted here.

Two Types of Mistakes

Not all errors are equal - one is far more costly

False Positive
Legit → Flagged

A real purchase is blocked. The customer calls the bank, card is unblocked, customer moves on.

Cost: minor inconvenience
False Negative  ⚠
Fraud → Missed

Real fraud slips through. The customer loses money. The bank pays chargebacks, loses trust, and faces regulatory scrutiny.

Cost: financial loss + reputation damage

These two types of mistakes have very different costs - and that shapes every decision in the project.

Feature Engineering

Turning Raw Data Into Fraud Signals

The Idea

The raw data doesn't speak fraud. We taught it how.

A timestamp is just a number. But 3 AM on a Wednesday tells a very different story.
🕑

Time of Day

A raw timestamp is meaningless to a model. We extracted hour, day of week, and month — so the model can learn that 3 AM is suspicious and noon is not.

2019-01-01 03:24:00
→
hour = 3  ⚠
📍

Distance to Merchant

The dataset has the cardholder's home coordinates and the merchant's location. We calculated the straight-line distance between them. A purchase 4,000 km from home is a major red flag. This became our top predictor.

lat/lon × 2 pairs
→
distance = 4,823 km  ⚠
👤

Customer Age

A date of birth means nothing on its own. We calculated each cardholder's age at the time of the transaction. Fraud rates differ significantly across age groups — older cardholders are targeted more often.

dob: 1968-03-14
→
age = 51
📈

Amount — Compressed Scale

Transaction amounts range from $1 to over $28,000. That extreme range confuses some models — a $28,000 purchase would dominate all other signals. Taking the logarithm brings everything into a manageable range without losing the pattern.

amount: $28,476
→
log_amt = 10.25

Categories → Yes / No Columns

A model cannot read the word "Online Shopping". So we created one column per category — each asking a yes/no question the model can actually process.

is_online_shopping? is_food_dining? is_entertainment? + 11 more

Same applied to 13 US states. This technique is called one-hot encoding.

Data Preparation

Splitting the Data — Before Any Model Sees It

The Rule

You can't test a student on questions they've already seen the answers to.

We locked away 20% of the data before any training happened — the model never saw it until the final exam.

The 80 / 20 Split

80% — Training Data  (271,685 transactions)
20% — Test
Training set

What the model learns from. It sees these transactions during training and adjusts itself to detect patterns.

Test set — locked away

The real exam. The model never sees these during training. All final results — Recall, F1, AUC — are measured here.

Fixing the Imbalance — SMOTE

Remember — only 0.52% of the training data is fraud. If the model trains on this as-is, it learns that saying "not fraud" is almost always correct. It never properly learns what fraud looks like.

SMOTE (Synthetic Minority Over-sampling Technique) generates new synthetic fraud examples by blending existing ones — until the training set is 50/50. The model now sees equal amounts of both.

0.52%
fraud before SMOTE
→
50%
fraud after SMOTE

⚠ SMOTE is applied only to the training set. The test set stays untouched — it must reflect real-world proportions.

Putting Features on Equal Footing

Some of our features have very different scales. Transaction amount ranges from $1 to $28,000. Age ranges from 18 to 90. If we feed both into a model as-is, the large-scale feature can drown out the smaller ones — not because it's more important, but purely because its numbers are bigger.

StandardScaler rescales all numeric features to have the same centre (mean = 0) and spread (std = 1). Every feature now competes on equal terms.

amount: $0 – $28,476
→
scaled: -1.2 to 3.4

⚠ The scaler is fitted on training data only — then applied to both sets. Fitting on all data would let test information leak into training, making results look better than they really are.
